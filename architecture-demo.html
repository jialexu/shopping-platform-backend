💾 Multi-Database Architecture

Demonstrates polyglot persistence - using the right database for the right job. Each service uses the database technology that best fits its data model and access patterns.

🐬 MySQL

Type: Relational (ACID)

Used By: Account Service, Payment Service

Why: Strong consistency for financial transactions and user data

Port: 3306

🐘 PostgreSQL

Type: Relational (Advanced)

Used By: Auth Service

Why: Advanced querying for token management and blacklisting

Port: 5432

🍃 MongoDB

Type: Document Store (NoSQL)

Used By: Item Service

Why: Flexible schema for product attributes and metadata

Port: 27017

💎 Cassandra

Type: Wide-Column Store (NoSQL)

Used By: Order Service, Inventory Service

Why: High write throughput for orders and inventory events

Port: 9042

🎯 Database Selection Strategy

  • MySQL: Traditional relational data with strict ACID requirements
  • PostgreSQL: Complex queries and advanced SQL features
  • MongoDB: Dynamic schemas and JSON-like documents
  • Cassandra: Massive write loads and time-series data
🔄 Data Flow Examples
User Registration: Client → Gateway → Account Service → MySQL (users table) Item Lookup: Client → Gateway → Item Service → MongoDB (items collection) Order Creation: Client → Gateway → Order Service → Cassandra (orders table) ↓ Inventory Service → Cassandra (inventory_by_sku table) Payment Processing: Client → Gateway → Payment Service → MySQL (payments table)

📨 Event-Driven Architecture with Kafka

Asynchronous communication between services using Apache Kafka. Events enable loose coupling and eventual consistency across the distributed system.

📬 Kafka Topics & Event Flow
Topic: order.created Publisher: Order Service Consumers: Inventory Service, Analytics Service Event: { orderId, userId, items[], timestamp } Topic: order.cancelled Publisher: Order Service Consumers: Inventory Service (restore stock), Payment Service (refund) Topic: payment.succeeded Publisher: Payment Service Consumers: Order Service (update status), Notification Service Event: { paymentId, orderId, amount, timestamp } Topic: payment.failed Publisher: Payment Service Consumers: Order Service (mark failed), Notification Service Event: { paymentId, orderId, reason, timestamp } Topic: payment.refunded Publisher: Payment Service Consumers: Order Service, Accounting Service Event: { paymentId, orderId, refundAmount, reason }

✨ Benefits of Event-Driven Architecture

  • Loose Coupling: Services don't need to know about each other
  • Scalability: Consumers can scale independently
  • Resilience: Events are persisted; consumers can catch up
  • Audit Trail: All events are logged in Kafka
  • Real-time Updates: Instant propagation across services

🔐 Security Architecture

JWT-based authentication with Spring Security OAuth2 Resource Server. Centralized authentication through Auth Service with token-based authorization at the Gateway level.

→

Extract & Validate JWT
⬇️
✓ Token Valid? → Forward to Service
✗ Token Invalid? → 401 Unauthorized
⬇️
3️⃣ Business Service
Process Request
JWT Token Structure: Header: { "alg": "HS256", "typ": "JWT" } Payload: { "sub": "user-id-123", "email": "user@demo.com", "iat": 1699999999, "exp": 1700086399 } Signature: HMACSHA256(base64Url(header) + "." + base64Url(payload), secret)

🔒 Security Features Implemented

  • JWT Authentication: Stateless token-based auth
  • Password Hashing: BCrypt with strength 10
  • Token Blacklist: Revoked tokens stored in PostgreSQL
  • CORS Configuration: Cross-origin request handling
  • Method Security: @PreAuthorize annotations
  • HTTPS Ready: SSL/TLS support for production

🐳 Docker Deployment Architecture

Complete containerized deployment with Docker Compose. One-command deployment with all services, databases, and infrastructure components.

docker-compose.yml Structure: services: ├── gateway (Spring Cloud Gateway) ├── auth-service (JWT Authentication) ├── account-service (User Management) ├── item-service (Product Catalog) ├── inventory-service (Stock Management) ├── order-service (Order Processing) ├── payment-service (Payment Processing) │ ├── mysql (Account + Payment DB) ├── postgres (Auth Token DB) ├── mongo (Item Catalog DB) ├── cassandra (Orders + Inventory DB) ├── cassandra-init (Schema Initialization) │ ├── zookeeper (Kafka Coordination) ├── kafka (Event Streaming) └── kafka-ui (Kafka Management UI)

🚀 Quick Start

docker-compose up -d

Starts all 15 containers in correct dependency order

🔍 Health Check

docker-compose ps

View status of all running containers

📊 View Logs

docker-compose logs -f

Stream logs from all services

🛑 Stop All

docker-compose down

Stop and remove all containers

📦 Container Configuration

  • Base Image: eclipse-temurin:21-jre (Java 21)
  • Build Tool: Maven 3.8+ with Spring Boot plugin
  • Networking: Bridge network for inter-container communication
  • Volumes: Database persistence and init scripts
  • Health Checks: Spring Actuator /health endpoints
  • Environment: SPRING_PROFILES_ACTIVE=docker
